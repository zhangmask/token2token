> **文档版本：** v2.0  
> **日期：** 2026-05-27  
> **状态：** 方案阶段  
> **定位：** 纯公益互助模式，不涉及金钱交易

---

## 一、项目概述

### 1.1 核心理念

**Token贡献度互助平台**是一个纯公益性质的社区，用户可以贡献自己用不完的Token额度给社区，同时获得贡献度；当用户需要使用Token时，可以用贡献度兑换其他用户贡献的额度。

### 1.2 关键特点

```
✅ 纯公益模式
├── 不涉及金钱交易
├── 不转售API Key
├── 社区互助性质
└── 贡献度作为激励机制

✅ 合规性
├── 不违反API提供商ToS（不转售）
├── 用户自愿贡献
├── 社区自治
└── 无商业行为

✅ 可持续性
├── 贡献度激励机制
├── 社区信任建设
├── 逐步扩大规模
└── 长期发展
```

### 1.3 与原方案的区别

| 维度 | 原方案（混合模式） | 新方案（纯互助） |
|------|-------------------|------------------|
| **交易模式** | 免费+付费 | 纯免费 |
| **金钱交易** | 有 | 无 |
| **法律风险** | 高 | 低 |
| **盈利模式** | 手续费 | 无/捐赠 |
| **可持续性** | 依赖盈利 | 依赖社区 |

## 三、贡献度机制设计

### 3.1 贡献度获取

```
贡献度获取规则：

1. 贡献Token
   ├── 公式：贡献度 = 贡献的Token数量 × 单价系数
   ├── 示例：贡献10000 tokens × $0.02/1000 = 0.2贡献度
   └── 说明：按市场标准价格换算

2. 邀请奖励
   ├── 被邀请用户首次贡献的10%
   ├── 示例：被邀请用户贡献100贡献度 → 邀请者获得10贡献度
   └── 限制：每个用户最多邀请100人

3. 社区贡献
   ├── 提交有效Bug报告：5贡献度
   ├── 提交代码改进：10-50贡献度
   ├── 帮助新用户：2贡献度/次
   └── 参与社区活动：按活动规则

4. 活动奖励
   ├── 新用户注册奖励：10贡献度
   ├── 连续登录奖励：每天1贡献度
   ├── 首次贡献奖励：20贡献度
   └── 节日活动奖励：按活动规则
```

### 3.2 贡献度使用

```
贡献度使用规则：

1. 兑换Token
   ├── 公式：可兑换Token = 贡献度 × 1000 ÷ 单价系数
   ├── 示例：1贡献度可兑换50000 tokens（假设单价$0.02/1000）
   └── 说明：贡献度直接兑换Token使用

2. 优先级提升
   ├── 贡献度越高，优先级越高
   ├── 高优先级用户优先获得Token
   └── 避免资源争抢

3. 兑换限制
   ├── 每日兑换上限：根据贡献度等级
   ├── 新用户：每日最多兑换100贡献度
   ├── 活跃用户：每日最多兑换500贡献度
   └── 核心用户：每日最多兑换1000贡献度
```

### 3.3 贡献度等级

```
等级体系：

┌─────────────┬─────────────┬─────────────┬─────────────┐
│   等级      │  贡献度要求 │  每日上限   │  特权       │
├─────────────┼─────────────┼─────────────┼─────────────┤
│  新手       │  0-100      │  100        │  基础功能   │
│  活跃       │  100-1000   │  500        │  优先匹配   │
│  核心       │  1000-5000  │  1000       │  专属客服   │
│  贡献者     │  5000+      │  2000       │  社区管理   │
└─────────────┴─────────────┴─────────────┴─────────────┘

升级条件：
├── 累计贡献度达到要求
├── 账号注册时间超过30天
├── 无违规记录
└── 社区评价良好
```

### 3.4 贡献度有效期

```
有效期规则：
├── 贡献度有效期：1年
├── 过期前30天提醒
├── 过期后自动失效
├── 可通过贡献续期
└── 活跃用户自动续期
```

## 五、安全设计

### 5.1 端到端加密

```
加密流程：
┌─────────────────────────────────────────────────────────────┐
│  1. 贡献者客户端                                            │
│     - 生成随机密钥K1（AES-256）                             │
│     - 使用K1加密API Key：EncryptedKey = AES(Key, K1)        │
│     - 使用平台公钥加密K1：EncryptedK1 = RSA(K1, PubKey)     │
│     - 发送：EncryptedKey + EncryptedK1                      │
│                                                             │
│  2. 平台服务器                                              │
│     - 存储：EncryptedKey + EncryptedK1                      │
│     - 平台无法解密原始Key                                   │
│                                                             │
│  3. 使用者客户端                                            │
│     - 接收：EncryptedKey + EncryptedK1                      │
│     - 使用用户私钥解密K1：K1 = RSA_Decrypt(EncryptedK1)     │
│     - 使用K1解密Key：Key = AES_Decrypt(EncryptedKey, K1)    │
│     - Key在内存中使用，使用后立即清除                        │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 Key安全保护

```
保护措施：
├── 使用次数限制
│   ├── 单Key每日最多100次调用
│   ├── 超过限制自动切换其他Key
│   └── 可配置（贡献者可调整）
│
├── 有效期限制
│   ├── Key默认24小时有效期
│   ├── 过期后自动失效
│   └── 可续期（需贡献者确认）
│
├── 使用监控
│   ├── 记录所有API调用
│   ├── 异常检测（大量请求、异常IP）
│   └── 自动封禁可疑行为
│
└── Key水印
    ├── 在Key中嵌入使用者ID
    ├── 泄露时可追溯
    └── 不影响Key正常使用
```

### 5.3 网络安全

```
安全措施：
├── HTTPS强制
│   ├── 所有通信使用TLS 1.3
│   ├── 证书固定（Certificate Pinning）
│   └── HSTS启用
│
├── API安全
│   ├── 请求签名（HMAC）
```

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,

    password_hash VARCHAR(255) NOT NULL,

    id UUID PRIMARY KEY,
```
