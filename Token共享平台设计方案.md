---

## 一、项目背景

### 1.1 问题描述

许多用户在订阅Coding Plan或API服务后，由于使用量不足，导致Token/额度在月底浪费。与此同时，另一些用户可能需要临时使用这些服务但不想承担完整的订阅费用。

### 1.2 解决方案

建立一个Token共享生态系统，让有剩余Token的用户可以贡献或出售给需要的人，实现资源的高效利用。

### 1.3 核心价值

- **资源优化**：减少Token浪费，提高资源利用率
- **成本降低**：让需要的用户以更低的成本使用API服务
- **社区共建**：建立Token共享社区，促进互利共赢

## 三、技术方案对比

### 3.1 三种实现方案

#### 方案一：中心化平台 + 本地代理（推荐）

```
架构：
贡献者A → 加密Key上传 → 平台存储
使用者B → 请求Token → 平台验证 → 发送加密Key给B的客户端
B的客户端 → 解密Key → 启动本地代理(localhost:端口)
B调用本地代理 → 代理转发到实际API

优点：
✅ 架构简单，易于实现和维护
✅ 平台完全控制Key的分发和使用
✅ 可以精确计量Token使用量
✅ 安全性高，Key不直接暴露

缺点：
❌ 平台是单点故障
❌ 需要开发跨平台客户端
❌ 平台需要承担Key存储的安全责任
```

#### 方案二：P2P直连模式

```
架构：
贡献者A → 运行本地代理服务 → 向平台注册服务地址
使用者B → 请求Token → 平台匹配 → 获取A的服务地址和密钥
B → 直接连接A的代理服务 → 通过加密通道调用API

优点：
✅ 去中心化，平台负载低
✅ 贡献者完全控制Key
✅ 无需平台存储Key

缺点：
❌ 贡献者必须在线才能提供服务
❌ 网络穿透问题（NAT、防火墙）
❌ 计量和计费更复杂
❌ 安全性依赖贡献者端
```

#### 方案三：混合模式（中心化+P2P）

```
架构：
普通Token → 中心化平台处理（方案一）
高价值/大额Token → P2P直连（方案二）
平台作为中间层，自动选择最优路径

优点：
✅ 灵活性最高，适应不同场景
✅ 可以优化成本和性能
✅ 用户体验统一

缺点：
❌ 架构复杂，开发成本高
❌ 需要维护两套系统
❌ 路由逻辑复杂
```

### 3.2 方案对比表

| 维度 | 方案一（中心化） | 方案二（P2P） | 方案三（混合） |
|------|------------------|---------------|----------------|
| **开发难度** | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **安全性** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| **性能** | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **可扩展性** | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **用户体验** | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| **平台负载** | 高 | 低 | 中等 |
| **适用场景** | 初期MVP | 技术社区 | 大规模平台 |

### 3.3 推荐方案

**推荐：方案一（中心化平台 + 本地代理）**

**理由：**
1. **快速验证**：可以快速上线MVP，验证商业模式
2. **安全性**：平台完全控制Key分发，降低风险
3. **可扩展性**：后续可以平滑升级到混合模式
4. **用户体验**：用户只需安装客户端，无需关心贡献者是否在线

## 五、安全设计

### 5.1 端到端加密（E2EE）

```
加密流程：
┌─────────────────────────────────────────────────────────────┐
│  1. 贡献者客户端                                            │
│     - 生成随机密钥K1（AES-256）                             │
│     - 使用K1加密API Key：EncryptedKey = AES(Key, K1)        │
│     - 使用平台公钥加密K1：EncryptedK1 = RSA(K1, PubKey)     │
│     - 发送：EncryptedKey + EncryptedK1                      │
│                                                             │
│  2. 平台服务器                                              │
│     - 存储：EncryptedKey + EncryptedK1                      │
│     - 平台无法解密原始Key                                   │
│                                                             │
│  3. 使用者客户端                                            │
│     - 接收：EncryptedKey + EncryptedK1                      │
│     - 使用用户私钥解密K1：K1 = RSA_Decrypt(EncryptedK1)     │
│     - 使用K1解密Key：Key = AES_Decrypt(EncryptedKey, K1)    │
│     - Key在内存中使用，使用后立即清除                        │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 Key安全保护

```
保护措施：
├── 使用次数限制
│   ├── 单Key每日最多100次调用
│   ├── 超过限制自动切换其他Key
│   └── 可配置（贡献者可调整）
│
├── 有效期限制
│   ├── Key默认24小时有效期
│   ├── 过期后自动失效
│   └── 可续期（需贡献者确认）
│
├── 使用监控
│   ├── 记录所有API调用
│   ├── 异常检测（大量请求、异常IP）
│   └── 自动封禁可疑行为
│
└── Key水印
    ├── 在Key中嵌入使用者ID
    ├── 泄露时可追溯
    └── 不影响Key正常使用
```

### 5.3 网络安全

```
安全措施：
├── HTTPS强制
│   ├── 所有通信使用TLS 1.3
│   ├── 证书固定（Certificate Pinning）
│   └── HSTS启用
│
├── API安全
│   ├── 请求签名（HMAC）
│   ├── 时间戳验证（防止重放）
│   ├── IP白名单（可选）
│   └── 速率限制
│
└── DDoS防护
    ├── 流量清洗
    ├── IP黑名单
    └── 自动限流
```

## 六、经济模型

### 6.1 货币体系

```
平台货币：
├── 美元（USD）
│   ├── 主要交易货币
│   ├── 充值/提现
│   └── 实时汇率
│
├── 贡献度（Contribution Score）
│   ├── 平台内积分
│   ├── 获取：贡献Token
│   ├── 使用：兑换Token、高级功能
│   └── 有效期：1年
│
└── 信用分（Credit Score）
    ├── 用户信用评估
    ├── 影响：交易限额、优先级
    └── 初始：100分
```

### 6.2 定价策略

```
定价规则：
├── 免费共享区
│   ├── 贡献者自愿提供
│   ├── 无价格限制
│   └── 获得贡献度奖励
│
├── 付费交易区
│   ├── 价格由贡献者设定
│   ├── 不能超过原价
│   ├── 最低价格限制（如原价的10%）
│   └── 价格调整需提前24小时通知
│
└── 平台手续费
    ├── 交易手续费：5-10%
    ├── 提现手续费：1-2%
    └── 贡献度兑换：免费
```

### 6.3 贡献度计算

```
贡献度获取规则：
├── 免费共享
│   ├── 贡献度 = 使用量（tokens）× 单价（美元/1000 tokens）÷ 1000
│   └── 示例：10000 tokens × $0.02 ÷ 1000 = 0.2贡献度
│
├── 付费出售
│   ├── 贡献度 = 售价 × 50%
│   └── 示例：售价$1.00 → 0.5贡献度
│
└── 邀请奖励
    ├── 被邀请用户首次消费的10%
    └── 示例：被邀请用户消费$10 → 1贡献度

贡献度使用规则：
├── 1贡献度 = 1美元额度
├── 可用于兑换任意Token
├── 不可转让、不可提现
└── 有效期1年
```

| 风险 | 影响 | 应对措施 |
